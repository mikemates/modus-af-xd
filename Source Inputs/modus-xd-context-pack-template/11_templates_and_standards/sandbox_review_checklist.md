# Sandbox Review Checklist

Use this checklist when Product, XD, Engineering, and QE review the Experience Sandbox before it becomes backlog input.

## Review Metadata

**Project:** [Name]

**Sandbox version:** [Branch/tag/commit]

**Review date:** [YYYY-MM-DD]

**Participants:** [Names]

**Outcome:** Approved / Approved with changes / Not ready

## Product and Experience Fit

- [ ] Sandbox reflects the current product intent.
- [ ] Primary user role and task are clear.
- [ ] Core flow solves the intended problem.
- [ ] Flow has a clear entry and completion point.
- [ ] Non-goals and missing areas are explicit.

## Flow and IA

- [ ] Major routes/screens are represented.
- [ ] Navigation model is understandable.
- [ ] Object/list/detail relationships are clear.
- [ ] Role/permission implications are visible or documented.
- [ ] Edge cases are represented or documented.

## States and Behavior

- [ ] Default states are represented.
- [ ] Loading states are represented or documented.
- [ ] Empty states are represented or documented.
- [ ] Error states are represented or documented.
- [ ] Success/confirmation states are represented or documented.
- [ ] Validation behavior is understandable.
- [ ] Recovery paths are clear.

## Components and Feasibility

- [ ] Component usage is realistic.
- [ ] Component library assumptions are documented.
- [ ] Custom components are flagged.
- [ ] Engineering feasibility concerns are captured.
- [ ] Reusable candidates are identified.
- [ ] Reference-only elements are identified.

## Content and Accessibility

- [ ] Key labels and messages are clear.
- [ ] Placeholder copy is marked.
- [ ] Terminology matches context files.
- [ ] Keyboard/focus expectations are documented.
- [ ] Accessibility risks are visible.

## Backlog Readiness

- [ ] Routes/states map to features, epics, or stories.
- [ ] Open questions are documented.
- [ ] Acceptance criteria implications are clear.
- [ ] Measurement/instrumentation implications are visible.
- [ ] Sandbox manifest is updated.
